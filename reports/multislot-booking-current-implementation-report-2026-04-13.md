## What â€śMultislot Bookingâ€ť Means In This Codebase

- `Listing` stores `totalSlots`, `availableSlots`, `bookingMode`, and `holdTtlMinutes`; `Booking` stores `slotsRequested`, `heldUntil`, `heldAt`, and `status`. That is the core data model supporting multislot requests and hold-based slot reservation. (`prisma/schema.prisma:120-126`, `prisma/schema.prisma:183-204`)
- `slotsRequested` defaults to `1`, is required at the database layer, and is constrained to `1..20` by both Zod validation and DB checks. (`src/lib/schemas.ts:321-332`, `src/lib/schemas.ts:362-373`, `prisma/migrations/20260325000000_protect_booking_integrity/migration.sql:27-39`)
- `bookingMode` is a listing-level switch between `"SHARED"` and `"WHOLE_UNIT"`. The database migration adds a check constraint limiting values to those two strings. (`prisma/schema.prisma:126`, `prisma/migrations/20260310100000_phase3_whole_unit_booking_mode/migration.sql:10-16`)

## Feature Flags And Gating

- Multislot booking requests are rejected in `createBooking` when `slotsRequested > 1` and `features.multiSlotBooking` is false. Single-slot bookings bypass that gate. (`src/app/actions/booking.ts:432-442`)
- Holds are rejected entirely when `features.softHoldsEnabled` is false. Because that getter only returns true for `ENABLE_SOFT_HOLDS=on`, both `off` and `drain` block new holds in `createHold`. Additional multislot hold requests are rejected when `slotsRequested > 1` and `features.multiSlotBooking` is false. (`src/app/actions/booking.ts:1005-1013`, `src/app/actions/booking.ts:1042-1052`, `src/lib/env.ts:502-507`)
- Whole-unit listing creation and listing updates are rejected when `bookingMode === "WHOLE_UNIT"` and `features.wholeUnitMode` is false. (`src/app/api/listings/route.ts:370-379`, `src/app/api/listings/[id]/route.ts:562-571`)
- Env validation enforces dependency rules at boot: whole-unit mode requires multi-slot booking, and `ENABLE_SOFT_HOLDS=on` also requires multi-slot booking. (`src/lib/env.ts:159-169`, `src/lib/env.ts:181-189`)

## Listing Configuration And Capacity Baseline

- New listings persist `totalSlots`, initialize `availableSlots` to the same value, and persist `bookingMode` with a default of `"SHARED"` if not provided. (`src/app/api/listings/route.ts:381-402`)
- Listing edits block `bookingMode` changes when future `ACCEPTED` bookings exist. The check does not count `PENDING` bookings. (`src/app/api/listings/[id]/route.ts:596-613`)
- Listing edits also block lowering `totalSlots` below the sum of future/non-expired committed demand that the code counts as `ACCEPTED` bookings with `endDate >= NOW()` plus active `HELD` bookings with `heldUntil > NOW()`. The code uses `SUM("slotsRequested")` for that committed count. (`src/app/api/listings/[id]/route.ts:615-631`)
- When `totalSlots` changes, the listing update recomputes `availableSlots` by adjusting the previous value by the slot delta and clamping the result into `0..totalSlots`. (`src/app/api/listings/[id]/route.ts:654-662`)

## Booking Creation (`createBooking`)

- `createBooking` validates auth, suspension, verified email, per-user rate limits, per-IP rate limits, and schema constraints before entering the transaction. (`src/app/actions/booking.ts:355-429`)
- Inside the transaction, it rejects exact-date duplicates for the same tenant/listing when an existing booking is `PENDING`, `ACCEPTED`, or `HELD`. This exact-date duplicate check does not filter `HELD` by `heldUntil`, so expired-but-unswept `HELD` rows can still block `createBooking`. (`src/app/actions/booking.ts:75-94`)
- It locks the listing row with `FOR UPDATE`, validates authoritative DB price, rejects owner self-booking, rejects block relationships, and rejects inactive listings. (`src/app/actions/booking.ts:96-165`)
- For whole-unit listings, `effectiveSlotsRequested` is forced to `listing.totalSlots` regardless of the client-provided `slotsRequested`. (`src/app/actions/booking.ts:167-171`)
- Capacity is checked with `SUM("slotsRequested")` across overlapping `ACCEPTED` bookings and overlapping active `HELD` bookings. `PENDING` bookings are not included in this particular capacity query. (`src/app/actions/booking.ts:173-194`)
- A second overlap check blocks the same tenant from having overlapping `PENDING`, `ACCEPTED`, or `HELD` bookings on the same listing. This overlap check also does not filter `HELD` by `heldUntil`, so expired-but-unswept `HELD` rows can still block `createBooking`. (`src/app/actions/booking.ts:196-216`)
- A successful `createBooking` call creates a `PENDING` booking and records `slotsRequested`, but it does not decrement `Listing.availableSlots` at creation time. Slot consumption for `PENDING -> ACCEPTED` happens later in `updateBookingStatus`. (`src/app/actions/booking.ts:231-252`, `src/app/actions/manage-booking.ts:267-365`)
- The idempotent and non-idempotent booking creation paths both call the same core transaction helper, but their outward error handling is not identical. The non-idempotent path retries serialization conflicts up to three times under `Serializable` isolation, while the idempotent wrapper has its own retry/caching/error-mapping behavior. (`src/lib/idempotency.ts:215-244`, `src/app/actions/booking.ts:454-549`, `src/app/actions/booking.ts:552-633`)

## Hold Creation (`createHold`)

- `createHold` validates auth, per-user/IP/per-listing rate limits, suspension, verified email, soft-hold enablement, schema constraints, and multislot feature flags before transaction entry. (`src/app/actions/booking.ts:941-1052`)
- Inside the transaction, it first enforces a maximum of `MAX_HOLDS_PER_USER = 3` active holds per user. (`src/app/actions/booking.ts:675-689`, `src/lib/hold-constants.ts:1-4`)
- It locks the listing row, validates price, blocks self-holds and blocked users, rejects inactive listings, and forces whole-unit requests to `listing.totalSlots`. (`src/app/actions/booking.ts:691-761`)
- Hold capacity is checked with `SUM("slotsRequested")` across overlapping `ACCEPTED` bookings and overlapping active `HELD` bookings. (`src/app/actions/booking.ts:763-786`)
- `createHold` also checks `listing.availableSlots` as a defense-in-depth guard before doing the conditional decrement. (`src/app/actions/booking.ts:788-791`)
- Duplicate hold/booking prevention for the same tenant/listing/date window covers `PENDING`, `HELD`, and `ACCEPTED`, but the `HELD` branch only blocks unexpired holds. (`src/app/actions/booking.ts:793-816`)
- Slot consumption for holds happens immediately via conditional `UPDATE "Listing" SET "availableSlots" = "availableSlots" - effectiveSlotsRequested ... AND "availableSlots" >= effectiveSlotsRequested`. (`src/app/actions/booking.ts:818-827`)
- A successful hold creates a `HELD` booking, writes `slotsRequested`, `heldUntil`, and `heldAt`, and uses per-listing TTL with fallback to the global default `HOLD_TTL_MINUTES = 15`. (`src/app/actions/booking.ts:841-868`, `src/lib/hold-constants.ts:1-2`)
- The idempotent and non-idempotent hold creation paths both call the same core transaction helper, but their outward error handling is not identical. The non-idempotent path retries serialization conflicts up to three times under `Serializable` isolation and has a dedicated `WHOLE_UNIT_OVERLAP` mapping, while the idempotent wrapper has its own retry/caching/error-mapping behavior. (`src/lib/idempotency.ts:215-244`, `src/app/actions/booking.ts:1062-1149`, `src/app/actions/booking.ts:1152-1218`)

## Booking State Machine And Slot Accounting

- The explicit allowed transitions are: `PENDING -> ACCEPTED|REJECTED|CANCELLED`, `ACCEPTED -> CANCELLED`, `HELD -> ACCEPTED|REJECTED|CANCELLED|EXPIRED`, and terminal states `REJECTED`, `CANCELLED`, and `EXPIRED`. (`src/lib/booking-state-machine.ts:19-26`)
- `updateBookingStatus` blocks manual transitions to `EXPIRED` before any other transition logic. (`src/app/actions/manage-booking.ts:42-47`)
- `updateBookingStatus` also performs inline expiry on read: if a loaded booking is already `HELD` and `heldUntil` is in the past, it tries to atomically mark the booking `EXPIRED`, restore slots with `LEAST`, write an audit row, and then returns `"This hold has expired."`. (`src/app/actions/manage-booking.ts:100-150`)
- `HELD -> ACCEPTED` does not decrement slots again because the slots were already consumed at hold creation. It only flips status to `ACCEPTED`, clears `heldUntil`, and increments `version` after re-locking the listing row. (`src/app/actions/manage-booking.ts:166-232`)
- `PENDING -> ACCEPTED` re-locks the listing row, recomputes `slotsNeeded` (`totalSlots` for whole-unit, otherwise `booking.slotsRequested`), verifies current `availableSlots`, reruns a `SUM("slotsRequested")` capacity query against overlapping `ACCEPTED` and active `HELD` bookings, flips the booking to `ACCEPTED`, and then decrements `Listing.availableSlots` conditionally. (`src/app/actions/manage-booking.ts:267-365`)
- In the `REJECTED` branch, slots are restored with `LEAST(..., "totalSlots")` only when the prior status was `HELD`. The same branch also handles `PENDING -> REJECTED`, but that path does not restore slots. (`src/app/actions/manage-booking.ts:458-508`)
- `ACCEPTED -> CANCELLED` and `HELD -> CANCELLED` both restore `booking.slotsRequested` with the same `LEAST(..., "totalSlots")` clamp. (`src/app/actions/manage-booking.ts:575-620`)
- Within `updateBookingStatus`, `PENDING -> CANCELLED` only flips booking status and writes audit data; it does not restore slots because `PENDING` does not consume them. Separate admin listing-removal/report-resolution flows also cancel `PENDING` bookings directly with `updateMany(...)`, outside this state-machine/audit path. (`src/app/actions/manage-booking.ts:635-675`, `src/app/actions/admin.ts:470-480`, `src/app/actions/admin.ts:741-757`)

## WHOLE_UNIT Specific Behavior

- Both booking creation and hold creation force whole-unit requests to use `listing.totalSlots` instead of the client-provided slot count. (`src/app/actions/booking.ts:167-171`, `src/app/actions/booking.ts:759-761`)
- `PENDING -> ACCEPTED` also recomputes whole-unit slot consumption from the currently locked listing row, not from stale booking input. (`src/app/actions/manage-booking.ts:296-303`)
- In the latest migration-defined database implementation in this repo, `check_whole_unit_overlap()` fires on `INSERT` or `UPDATE` when `NEW.status IN ('ACCEPTED', 'HELD')`, locks the listing row, and blocks overlapping bookings for whole-unit listings. (`prisma/migrations/20260328000000_fix_whole_unit_held_check/migration.sql:6-48`)
- The earlier March 10 soft-holds migration checked unexpired `HELD` rows with `heldUntil > NOW()`, but the later March 28 replacement counts any overlapping `HELD` row and does not filter by `heldUntil`. That difference is real in the migration history and should not be collapsed into a single claim. (`prisma/migrations/20260310200001_phase4_soft_holds_indexes/migration.sql:34-47`, `prisma/migrations/20260328000000_fix_whole_unit_held_check/migration.sql:24-35`)

## Search And Availability Behavior

- Search v2 conditionally chooses between SearchDoc queries and legacy `data.ts` queries using `isSearchDocEnabled(...)`, which reads `ENABLE_SEARCH_DOC` and allows URL overrides only outside production. (`src/lib/search/search-v2-service.ts:206-210`, `src/lib/search/search-doc-queries.ts:1526-1540`)
- In the non-semantic SearchDoc path, the base WHERE conditions subtract active `HELD` slots from `d.available_slots` when soft holds are enabled or draining. The same WHERE-clause subtraction logic is duplicated in the search facets route. The selected `availableSlots` payload remains the raw stored `available_slots` column, not an adjusted value. (`src/lib/search/search-doc-queries.ts:455-470`, `src/app/api/search/facets/route.ts:157-172`, `src/lib/search/search-doc-queries.ts:808-824`, `src/lib/search/search-doc-queries.ts:1026-1034`, `src/lib/search/search-doc-queries.ts:1218-1226`, `src/lib/search/search-doc-queries.ts:1400-1408`)
- The SearchDoc semantic path is different: `semanticSearchQuery()` passes `minAvailableSlots` into the `search_listings_semantic(...)` SQL function, and the function filters on raw `sd.available_slots >= ...` without subtracting active `HELD` slots. (`src/lib/search/search-doc-queries.ts:1594-1667`, `prisma/migrations/20260314200000_fix_semantic_price_cast/migration.sql:79-97`)
- In the legacy `data.ts` path, availability checks use only `l."availableSlots" >= $1`; they do not subtract active `HELD` slots in the query itself. (`src/lib/data.ts:351-360`, `src/lib/data.ts:627-635`)
- Because of that split, it would be inaccurate to say â€śsearch always subtracts held slots.â€ť The code subtracts held slots in the non-semantic SearchDoc/facets WHERE clause, but not in the legacy `data.ts` SQL path or the semantic SearchDoc SQL function. (`src/lib/search/search-v2-service.ts:345-366`, `src/lib/data.ts:351-360`, `src/lib/search/search-doc-queries.ts:455-470`, `src/lib/search/search-doc-queries.ts:1594-1667`, `prisma/migrations/20260314200000_fix_semantic_price_cast/migration.sql:79-97`)
- Separately, `getListingAvailability()` computes `effectiveAvailable = availableSlots + ghostHolds`, where `ghostHolds` is the sum of expired `HELD` bookings that have not yet been swept. That helper exists, but this report does not claim which pages currently call it. (`src/lib/listing-availability.ts:11-41`)

## Cron Repair And Audit Paths

- The expired-holds sweeper runs when soft holds are enabled or draining, uses an advisory transaction lock, selects expired `HELD` bookings with `FOR UPDATE OF b SKIP LOCKED`, marks each selected row as `EXPIRED`, restores `slotsRequested` to the listing with `LEAST`, writes booking audit rows, sends notifications outside the transaction, and marks affected listings dirty for SearchDoc refresh. The selection query inner-joins tenant and owner users, so it is not literally â€śall expired HELD rowsâ€ť if a row no longer matches those joins. (`src/app/api/cron/sweep-expired-holds/route.ts:47-55`, `src/app/api/cron/sweep-expired-holds/route.ts:71-147`, `src/app/api/cron/sweep-expired-holds/route.ts:172-208`, `prisma/schema.prisma:196-210`, `prisma/migrations/20260325000000_protect_booking_integrity/migration.sql:18-25`)
- The reconcile-slots cron does not depend on the booking-audit feature flag. For listings where `l.status = 'ACTIVE'`, it computes `expected` as `totalSlots - SUM(slotsRequested)` using currently active `ACCEPTED` bookings whose date range overlaps `NOW()` plus active `HELD` bookings. (`src/app/api/cron/reconcile-slots/route.ts:32-34`, `src/app/api/cron/reconcile-slots/route.ts:48-69`)
- When drift is small enough, reconcile-slots auto-fixes `availableSlots` with `GREATEST(0, expected)` and marks those listings dirty for SearchDoc refresh. This cron updates `Listing.availableSlots`; it does not rewrite or expire `Booking` rows. (`src/app/api/cron/reconcile-slots/route.ts:85-92`, `src/app/api/cron/reconcile-slots/route.ts:129-132`)
- Booking audit rows are always written inside the parent transaction and are not feature-gated in `logBookingAudit()` itself. The helper strips common PII keys from `details`. (`src/lib/booking-audit.ts:43-74`)
