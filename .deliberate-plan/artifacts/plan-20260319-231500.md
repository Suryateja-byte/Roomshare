After:
```typescript

    if (process.env.NODE_ENV === "production") {
      logger.sync.error("[Turnstile] CAPTCHA disabled in production — bot protection inactive", {
        turnstileEnabled: process.env.TURNSTILE_ENABLED,
```

- Users can still log in/register during the outage window

**Test plan**:

2. Unit: When disabled in development, no error logged
3. Unit: When enabled, no warning logged (normal path)

## Fix 12: Admin Write/Destructive Action Rate Limiting (REDESIGNED)

**Priority**: P2 | **Risk**: MEDIUM | **Effort**: 25 min | **Confidence**: 93%

**REDESIGNED**: Rate-limit only write/destructive actions (6 functions). Skip reads (4 functions) — admin dashboard loads getUsers + getListingsForAdmin + getReports + getAdminStats on mount; rate-limiting reads would break the dashboard.

**File**: `src/app/actions/admin.ts`

### New imports (add at top):
```typescript
import { checkRateLimit, RATE_LIMITS, getClientIPFromHeaders } from "@/lib/rate-limit";
import { headers } from "next/headers";
```

### New rate limit configs in `src/lib/rate-limit.ts` (add to RATE_LIMITS object):
```typescript
  adminWrite: { limit: 20, windowMs: 60_000 },      // 20 writes/min per admin
  adminDelete: { limit: 5, windowMs: 3_600_000 },    // 5 destructive ops/hour per admin
```

### Functions to rate-limit (6 write/destructive actions):

| Function | Line | Config | Reason |
|----------|------|--------|--------|
| `toggleUserAdmin` | 127 | adminWrite | Privilege escalation |
| `suspendUser` | 180 | adminWrite | User impact |
| `updateListingStatus` | 321 | adminWrite | Content moderation |
| `deleteListing` | 376 | adminDelete | Destructive |
| `resolveReport` | 545 | adminWrite | Moderation |
| `resolveReportAndRemoveListing` | 605 | adminDelete | Destructive |

### Functions NOT rate-limited (4 read actions):

| Function | Line | Reason |
|----------|------|--------|
| `getUsers` | 43 | Dashboard load, already behind requireAdmin |
| `getListingsForAdmin` | 236 | Dashboard load |
| `getReports` | 472 | Dashboard load |
| `getAdminStats` | 759 | Dashboard load, fires 9 parallel COUNTs |

### Pattern for each write function (add after requireAdmin check):
```typescript
  const adminCheck = await requireAdmin();
  if (adminCheck.error) return adminCheck;

  // Rate limit admin writes
  const headersList = await headers();
  const ip = getClientIPFromHeaders(headersList);
  const rl = await checkRateLimit(
    `${ip}:${adminCheck.userId}`,
    "adminWrite",
    RATE_LIMITS.adminWrite
  );
  if (!rl.success) {
    return { error: "Too many requests. Please slow down." };
  }
```

Use `adminDelete` config for `deleteListing` and `resolveReportAndRemoveListing`.

**Test plan**:
1. Unit: Write action with rate limit exceeded returns error
2. Unit: Read action (getUsers) has NO rate limit check
3. Unit: Delete action uses stricter adminDelete config
4. Unit: Rate limit identifier includes userId (per-admin, not per-IP)

**Rollback**: Remove rate limit imports and checks from each function. Remove RATE_LIMITS entries.
