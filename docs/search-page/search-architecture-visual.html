SSR Data Flow Request Lifecycle

What happens when a user navigates to /search?q=Austin&minPrice=500

sequenceDiagram
  participant U as User Browser
  participant P as page.tsx (SSR)
  participant V2 as search-v2-service
  participant CB as CircuitBreaker
  participant SDQ as search-doc-queries
  participant DB as PostgreSQL
  participant CL as Client Hydration

  U->>P: GET /search?q=Austin&minPrice=500
  P->>P: parseSearchParams(rawParams)
  P->>P: Check boundsRequired (q without bounds?)
  Note over P: q=Austin has lat/lng → bounds derived

  P->>CB: circuitBreakers.searchV2.execute()
  alt Circuit CLOSED (normal)
    CB->>V2: executeSearchV2({rawParams, limit:12})
    V2->>V2: parseSearchParams → filterParams
    V2->>V2: Detect sort, cursor, keyset mode
    V2->>SDQ: getSearchDocListingsFirstPage(params)
    SDQ->>SDQ: buildSearchDocWhereConditions()
    SDQ->>SDQ: buildOrderByClause()
    SDQ->>DB: SELECT ... FROM listing_search_docs WHERE ... LIMIT 13
    DB-->>SDQ: 13 rows (12 + 1 for hasNextPage)
    SDQ->>SDQ: mapRawListingsToPublic()
    SDQ->>SDQ: buildCursorFromRow(lastRow)
    SDQ-->>V2: {items: 12, nextCursor, total, hasNextPage}
    V2-->>CB: result
    CB-->>P: result (record success)
  else Circuit OPEN (V2 down)
    CB-->>P: CircuitOpenError (instant, no 10s wait)
    P->>P: logger.sync.info("circuit open")
    P->>P: Fall through to V1
  end

  P->>P: Build searchParamsString (strip cursor/page)
  P->>P: Extract initialNextCursor

  P-->>U: HTML with SSR listings + hydration payload
  U->>CL: React hydrates SearchResultsClient
  CL->>CL: Initialize seenIdsRef with SSR listing IDs
  CL->>CL: Fetch /api/favorites?ids=...

V2 / V1 Fallback Chain Resilience

Triple fallback: V2 SearchDoc → V1 Prisma joins → Error boundary

flowchart TD
  Start([User navigates to /search]) --> FF{Feature flag
searchV2 enabled?}
  FF -->|No| V1Path
  FF -->|Yes| CB{Circuit breaker
searchV2 state?}

  CB -->|OPEN| CBSkip["⚡ Skip V2 instantly
No 10s timeout penalty"]
  CBSkip --> V1Path
  CB -->|CLOSED/HALF_OPEN| V2Try

  V2Try["Try V2: executeSearchV2()"] --> TO{Timeout
10s exceeded?}
  TO -->|Yes| V2Fail["V2 Timeout
Sentry.captureException"]
  TO -->|No| V2Check{V2 returned
valid data?}

  V2Check -->|Yes| V2OK["✅ Use V2 data
SearchDoc + keyset cursor"]
  V2Check -->|No/Error| V2Fail
  V2Fail --> V1Path

  V1Path["V1: getListingsPaginated()"] --> V1TO{Timeout
10s exceeded?}
  V1TO -->|No| V1OK["✅ Use V1 data
Multi-table JOIN"]
  V1TO -->|Yes| V1Err["💥 Error thrown"]
  V1Err --> EB["error.tsx boundary
Try again + Go home"]

  V2OK --> Render["Render SearchResultsClient
+ V1PathResetSetter or V2MapDataSetter"]
  V1OK --> Render

"Load More" Pagination Flow Client-Side

Cursor-based infinite scroll with deduplication, 60-item cap, and degradation handling

flowchart TD
  Click["User clicks 'Show more places'"] --> Guard{nextCursor exists?
Not already loading?}
  Guard -->|No| Stop["Do nothing"]
  Guard -->|Yes| SA["Server Action: fetchMoreListings(cursor, rawParams)"]

  SA --> RL{Rate limit
check}
  RL -->|Blocked| Err429["Throw 'Rate limited'"]
  RL -->|OK| CB3{Circuit breaker
searchV2}

  CB3 -->|OPEN| V1Fall["V1 fallback:
return {degraded: true}"]
  CB3 -->|OK| V2Call["V2: executeSearchV2 with cursor"]

  V2Call --> V2OK{V2 succeeded?}
  V2OK -->|Yes| Return["Return {items, nextCursor, hasNextPage}"]
  V2OK -->|No| V1Fall

  Return --> Client["SearchResultsClient"]
  V1Fall --> Client

  Client --> DegCheck{result.degraded?}
  DegCheck -->|Yes| ShowMsg["Show error:
'Can't load more right now'
Remove button"]
  DegCheck -->|No| Dedup["Filter by seenIdsRef Set
O(1) lookup per item"]

  Dedup --> Append["Append to extraListings"]
  Append --> CapCheck{allListings.length
≥ 60?}
  CapCheck -->|Yes| CapMsg["Show: 'Refine your filters'
Hide button"]
  CapCheck -->|No| Ready["Show 'Show more places'
with next cursor"]

  Err429 --> FriendlyErr["'Too many requests —
wait 30 seconds'"]
