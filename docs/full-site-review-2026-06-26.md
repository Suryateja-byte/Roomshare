# Roomshare — Full-Site Review (2026-06-26)

> Multi-agent code review: 16 specialized reviewers across the whole app (35 pages, 61 API routes, 192 components), every finding adversarially re-verified against the cited code. 74 raw findings -> **64 confirmed**, 10 rejected as false positives.

**Severity tally:** 🟧 4 high · 🟨 16 medium · ⬜ 38 low · 🔵 6 info

## Executive summary

Roomshare's core mechanics (server-side state validation, Stripe idempotency keys, advisory locks, dedup constraints) are largely sound, but the review surfaced two genuinely serious correctness gaps plus broad systemic weaknesses. First, the owner DELETE path on listings (src/app/api/listings/[id]/route.ts:415-464) is the one listing mutation that skips the moderation write-lock every other path enforces, so a host can hard-delete an admin-paused/under-investigation listing and destroy the evidence — a direct hit on the project's #1 priority (abuse-resistance). Second, the banned-user auto-refund queue has no stale-PROCESSING reclaim (src/lib/payments/refund-queue.ts), so a worker crash after the Stripe refund call permanently strands the row, leaving money moved at Stripe but never reconciled locally. Beyond these, the money/idempotency paths are dangerously under-tested (contact-credit deduction, webhook replay, double-click defense, and concurrency all lack assertions), and there are recurring lower-severity patterns around IP-only rate limiting, raw-userId/PII logging, and overlay accessibility. Net: ship-blocking fixes are narrow and well-localized; the larger risk is a thin test net under the revenue and moderation flows.

## Overall assessment

Healthy foundation with a few sharp edges. The architecture is disciplined — UI stays out of the DB, state transitions are server-validated, payments use deterministic Stripe idempotency keys and advisory locks, and dedup is backed by real unique constraints. There are no confirmed auth bypasses, IDOR/PII-exposure holes, or double-charge bugs in production paths. The two high-severity items are real but bounded (each requires a legitimate authenticated actor or a narrow crash window) and are localized one-line-to-one-handler fixes. The dominant systemic weakness is test coverage on the money and idempotency flows: several core deduction, replay, and concurrency guards exist and read as correct but have zero regression coverage, so a future refactor could silently reintroduce a revenue-leak or double-grant bug. The long tail of low/info findings (IP-only rate limiting, raw-userId logging, dialog a11y, dead/duplicated code) reflects convention drift rather than active defects. Verdict: not yet launch-clean, but a short, high-leverage punch list closes the meaningful gaps.

## Top risks (ranked by real-world impact)

### 1. 🟧 Owner DELETE bypasses the moderation write-lock and hard-deletes admin-paused listings
*Area:* listings-crud / reliability-state (src/app/api/listings/[id]/route.ts:415-464)  ·  *Severity:* **high**

**Why it matters:** Every other listing mutation (PATCH profile/availability, status changes) gates on getHostModerationWriteLockResult and returns 423 when statusReason is ADMIN_PAUSED or SUPPRESSED. The DELETE transaction does not: it selects only ownerId/images/version, then branches solely on report.count. An admin PAUSE sets statusReason='ADMIN_PAUSED' without creating a Report row, so reportCount===0 routes to hard-delete (tombstone canonical inventory + listing.delete). A host whose listing was frozen for abuse investigation can simply DELETE it, destroying the listing and its evidence and defeating the moderation hold — a broken abuse-resistance/auditability invariant, the project's top priority. This is the same defect reported twice across two review areas.

**Fix:** In the DELETE FOR UPDATE transaction, add statusReason to the SELECT, then call getHostModerationWriteLockResult({ statusReason, moderationWriteLocksEnabled: features.moderationWriteLocks }) and return 423 LISTING_LOCKED before suppress/delete (admins take a separate path). The host-side lock fires regardless of the feature flag for ADMIN_PAUSED/SUPPRESSED, so the guard is flag-independent.

### 2. 🟧 Banned-user auto-refund rows can be stranded in PROCESSING forever after a worker crash
*Area:* cron-jobs / payments (src/lib/payments/refund-queue.ts:266-432)  ·  *Severity:* **high**

**Why it matters:** processRefundQueueRow claims a row (PROCESSING), calls Stripe refunds.create (money moves), then commits the local Refund record in a SEPARATE transaction. A crash/timeout/SIGKILL between the Stripe call and the DB commit leaves the row PROCESSING permanently: the claim query only ever selects PENDING, releaseUnprocessedRows only handles the in-process time-budget break (not a crash), and unlike the outbox drain (5-min IN_FLIGHT reclaim) and embeddings (recoverStuckEmbeddings) there is no reaper. Result: Stripe refunded the banned user but the system never records it, Payment.autoRefundStatus stays stuck, and nothing retries — a permanent reconciliation/audit gap on a money path. Reported twice as the same root cause.

**Fix:** Add a stale-PROCESSING reaper at the top of processRefundQueueOnce: UPDATE refund_queue_items SET status='PENDING', attempt_count=GREATEST(attempt_count-1,0) WHERE status='PROCESSING' AND updated_at < NOW() - INTERVAL '5 minutes'. The deterministic idempotencyKey (auto-refund:{paymentId}:{reason}) makes re-claim safe — Stripe will not double-refund — so the reaper is the missing half of the at-least-once contract.

### 3. 🟧 Core money/abuse flows (credit deduction, webhook replay, double-click, concurrency) have no regression tests
*Area:* testing-gaps (contact-paywall.test.ts, webhook-worker.test.ts, stripe-webhook-route.test.ts; no real-DB integration test)  ·  *Severity:* **high**

**Why it matters:** The actual one-credit deduction paths (FREE/PASS/PACK createConsumption) are never asserted — every create assertion in the paywall test is .not.toHaveBeenCalled — so a bug consuming 0 credits (free contacts / revenue leak) or 2 credits (overcharge), or selecting an exhausted PACK grant, would pass CI. Likewise, webhook entitlement-grant idempotency (processedAt guard + existing-grant guard), the clientIdempotencyKey double-click defense, the P2002 concurrent-capture recovery branch, and any real-DB concurrency test for double-spend are all unexercised. The guards currently read as correct, but CLAUDE.md makes idempotency/replay/double-click tests a non-negotiable for hold/grant lifecycle logic, and this is the top-priority revenue path with no safety net against future regressions.

**Fix:** Add targeted tests: (1) each consume path asserts create called with the right source and grant; (2) processedAt-set and existing-grant cases assert no second grant; (3) split the dedup test into idempotency-key-match vs unit-identity-match using mockResolvedValueOnce; (4) P2002 capture race returns 200 with no double outbox enqueue; (5) one pglite integration test running two concurrent consumeContactEntitlement calls (same idempotency key, and same unit identity) asserting exactly one consumption.

### 4. 🟨 PaymentIntent can attach to the wrong pending Payment row via metadata-only fallback
*Area:* reliability-state / payments (src/lib/payments/webhook-worker.ts:64-92, checkout-session-status.ts:158-185)  ·  *Severity:* **medium**

**Why it matters:** When no Payment matches by stripePaymentIntentId, the fallback picks the most-recent pending row whose metadata 'matches' — but matchesCheckoutMetadata compares only paywall fields, not the Stripe checkout session id or client idempotency key. Two checkout sessions for the same listing+product (double-click / abandoned-then-retried) produce two CHECKOUT_CREATED rows with identical metadata and null intent id; with out-of-order event processing the succeeded intent can bind to the wrong row. The legitimate session's later completed event then collides on the @unique stripePaymentIntentId, permanently FAILs the webhook, and corrupts Payment-to-charge attribution (refund/dispute handlers resolve by intent id). No money loss, but a stuck event and broken reconciliation invariant.

**Fix:** Constrain the fallback: match on stripeCheckoutSessionId (or carry a paymentId/sessionId in PaymentIntent metadata). If more than one row matches, refuse to bind and create a fresh Payment keyed solely by stripePaymentIntentId so the @unique constraint enforces 1:1.

### 5. 🟨 Email provider error bodies echoing recipient addresses are logged unredacted via raw console.*
*Area:* pii-secrets (src/lib/email.ts:126,141)  ·  *Severity:* **medium**

**Why it matters:** sendEmail logs the raw Resend response body and exceptions with console.error/console.warn, bypassing the logger's redactSensitive layer. Resend validation errors echo the offending recipient email (real user PII passed as `to`), so recipient addresses land unredacted in the platform log stream on failure paths — a direct violation of the #1 non-negotiable (no raw PII in logs). The sibling sendNotificationEmailWithPreference already uses hashIdForLog, confirming this is an oversight.

**Fix:** Route all email-failure logging through logger.sync.error with sanitizeErrorMessage/redactSensitive applied to the body; never log the recipient — use hashIdForLog(to) if a correlation handle is needed.

### 6. 🟨 New-message email notifications have no per-recipient throttle (inbox-flood / harassment vector)
*Area:* messaging (src/lib/messaging/send-conversation-message.ts:201-225)  ·  *Severity:* **medium**

**Why it matters:** Every sent message fires an email to each other participant with no cooldown, dedup, batching, or online-suppression — the only gate is the recipient's global on/off preference. The sender limit is 100 msgs/hour, so one sender can deliver ~100 emails/hour to a single recipient inside an existing (paywall-paid, persistent) thread, and a harasser can flood before being blocked. For a trust-and-safety-first rental app this is an abuse-resistance gap, not polish.

**Fix:** Add a per-(conversation,recipient) email cooldown/digest (track lastMessageEmailedAt or a short KV key) and/or suppress the email when the recipient is currently active in the thread; keep the in-app notification per-message.

### 7. ⬜ Authenticated LLM endpoints (chat, agent) and several abuse buckets are rate-limited by IP only
*Area:* rate-limit-abuse / authz-api (chat/route.ts:290-291, agent/route.ts:57; reports/reviews/favorites)  ·  *Severity:* **low**

**Why it matters:** Chat/agent require auth but key the limiter on getClientIP alone, while the rest of the codebase consistently uses ${ip}:${userId}. IP-only keying lets a single authenticated user multiply paid Groq spend by rotating IPs, and collectively throttles many users behind one NAT. createReport/createReview/toggleFavorite share the IP-only pattern, weakening spray-across-targets abuse protection against IPv6/NAT.

**Fix:** Key authenticated abuse-sensitive limiters on ${ip}:${userId} (matching phone-reveal/messages), and normalize IPv6 identifiers to the /64 prefix in getClientIP. Leave unauthenticated *ByIp limits as-is (no user id pre-auth).

### 8. 🟨 Concierge chat panel lacks dialog semantics, focus trap, and focus return
*Area:* accessibility (src/components/NeighborhoodChat.tsx:811-827)  ·  *Severity:* **medium**

**Why it matters:** The most interactive surface on the listing page opens as a plain z-9999 overlay with no role=dialog, aria-modal, focus trap, or focus-return-to-FAB. Keyboard/SR users can Tab into the still-live page behind it, AT never announces a dialog opened, and focus is lost on close — a WCAG 2.4.3/4.1.2 gap, inconsistent with every other overlay (FilterModal, MobileSearchOverlay) that already uses the in-house FocusTrap.

**Fix:** Wrap the open panel in the existing <FocusTrap active returnFocus> and add role=dialog, aria-modal=true, and aria-label/aria-labelledby — matching the established pattern.

## Systemic themes

**1. AuthZ/state guards applied inconsistently across mutation paths**  
Protective checks exist and are enforced on most paths but are silently missing on one sibling, so an attacker/host takes the unguarded door. The clearest case: every listing mutation gates on the moderation write-lock except owner DELETE, which then hard-deletes moderation-held content. The dead authorized() callback (no middleware/proxy wiring) is a related shape — intended edge-level gating is absent and the platform is safe only because per-handler checks happen to cover it.  
_Affected:_ `src/app/api/listings/[id]/route.ts (DELETE vs PATCH write-lock)`, `src/auth.ts authorized() callback (dead, not wired into proxy.ts)`, `src/app/api/test-helpers/route.ts (ownership-agnostic destructive actions gated only by env)`

**2. Money/idempotency paths are correct-but-fragile: side-effect-before-commit with no crash recovery**  
Several money-moving flows do the external/irreversible action first and persist the local record in a separate transaction, relying on a deterministic idempotency key as the only safety net — and lack a reaper for crash-stranded rows. Correctness today depends on invariants (key composition, claim-status filters) that no test pins, so a refactor could turn a stuck row into a double-write.  
_Affected:_ `src/lib/payments/refund-queue.ts (no stale-PROCESSING reclaim; Stripe call before local record)`, `src/lib/payments/webhook-worker.ts (metadata-only fallback; unbounded banned-user enqueue)`, `src/lib/outbox/drain.ts (handler commits before COMPLETED write → duplicate cache_invalidations)`

**3. Thin test net under the revenue, moderation, and concurrency surfaces**  
The most business-critical guards — contact-credit deduction, webhook replay/grant idempotency, double-click defense, P2002 concurrent capture, real-DB double-spend — exist and read correctly but have no regression coverage, despite CLAUDE.md mandating idempotency/abuse tests for these exact flows. CI would not catch a revenue-leak or double-grant regression.  
_Affected:_ `src/__tests__/lib/payments/contact-paywall.test.ts`, `src/__tests__/lib/payments/webhook-worker.test.ts`, `src/__tests__/api/stripe-webhook-route.test.ts`, `missing pglite integration test for consume/grant concurrency`

**4. PII/ID hygiene relies on convention, not enforcement**  
The redaction layer only scrubs a fixed key set and value patterns; raw user ids and provider error bodies routed outside logger (or as top-level fields) survive. hashIdForLog is the established convention but is applied unevenly, so the boundary is enforced by discipline rather than a chokepoint.  
_Affected:_ `src/lib/email.ts (raw console.* with recipient PII)`, `src/lib/logger.ts (top-level userId never redacted)`, `src/auth.ts and src/lib/api-error-handler.ts (raw userId in meta)`

**5. Rate-limit identifiers keyed by IP where a stronger authenticated key exists**  
Authenticated, cost- or abuse-sensitive endpoints key on client IP only while the codebase's own stronger pattern (${ip}:${userId}, /64 IPv6 normalization) is available and used elsewhere — enabling IP-rotation cost multiplication and shared-NAT collateral throttling.  
_Affected:_ `src/app/api/chat/route.ts and agent/route.ts (paid LLM)`, `src/app/api/reports|reviews|favorites/route.ts`, `src/app/api/messages/route.ts POST (no pre-auth IP throttle, unlike GET)`

**6. Overlay/dialog accessibility and live-region gaps**  
Interactive overlays and trust-and-safety forms miss dialog semantics, focus management, programmatic labels, and status announcements — handled correctly elsewhere in the app, so these are consistency regressions rather than unknown patterns.  
_Affected:_ `src/components/NeighborhoodChat.tsx (no dialog/focus trap, no aria-expanded on FAB)`, `src/components/ReportButton.tsx (Select label not wired; success/error not announced)`, `src/components/messages/MessageComposer.tsx (counter announces every keystroke)`

**7. Dead, duplicated, and drifted code that misleads the security/behavior model**  
Parallel implementations and orphaned wiring create false confidence and divergence risk: in-memory filters tested but never run in prod, an orphaned embeddings-recovery cron with a live producer, duplicate authorized callbacks, and schema-vs-migration index drift.  
_Affected:_ `src/lib/data.ts (in-memory filters/sort, test-only)`, `src/app/api/cron/embeddings-maintenance (never dispatched while producer is live)`, `prisma/schema.prisma vs migrations (partial-index drift on entitlement idempotency)`

## Quick wins

- Add the moderation write-lock guard to the owner DELETE handler (select statusReason in the FOR UPDATE, call getHostModerationWriteLockResult, return 423) — closes the highest-priority abuse/auditability hole in a few lines.
- Add a stale-PROCESSING reaper to refund-queue (UPDATE ... SET status='PENDING' WHERE status='PROCESSING' AND updated_at < NOW() - INTERVAL '5 minutes') — the idempotency key already makes re-claim safe.
- Route src/lib/email.ts failure logging through logger with sanitizeErrorMessage/hashIdForLog instead of raw console.* — stops recipient-email PII leaking into platform logs.
- Clamp client-controlled limits with one line (Math.min(Math.max(Math.trunc(Number(limit)||N),1),100)) on the notifications and getRecentlyViewed/admin server actions to kill the resource-exhaustion vector.
- Key the chat/agent (and report/review/favorite) rate limiters on ${ip}:${userId} to match the rest of the codebase and bound per-account LLM/abuse cost.
- Wrap the Concierge chat panel in the existing <FocusTrap returnFocus> and add role=dialog/aria-modal — reuses an in-house component, gives correct keyboard/SR behavior for free.
- Add id="reason" to the ReportButton SelectTrigger and wrap its success/error messages in role=status/role=alert — two small WCAG fixes on a trust-and-safety form.
- Add the consume-path and webhook-idempotency unit tests (assert create called with correct source/grant; assert no second grant on processedAt-set/existing-grant) — cheap regression coverage on the revenue path.
- Pin an explicit Stripe apiVersion (and the webhook endpoint version) so typed field reads used in amount/grant math can't drift on a Stripe-side default bump.
- Add deletedAt: null to markConversationMessagesAsReadForUser to match every other messaging predicate (one-line consistency fix).

## All confirmed findings (64)


### 🟧 HIGH

- **Auto-refund queue has no stale-PROCESSING reclaim: a crash after the Stripe refund call permanently strands the row**  
  `src/lib/payments/refund-queue.ts:266-289 (crash window), 292-305 (releaseUnprocessedRows only covers in-process time-budget break), 368-399 (claim query selects only status='PENDING')` · _cron-jobs / idempotency_
  - processRefundQueueRow claims the row (status PROCESSING) in one transaction, calls getStripeClient().refunds.create(...) — which actually moves money — then commits recordQueuedRefund in a SEPARATE transaction. If the function crashes or hits Vercel maxDuration between the Stripe call (line 271) and the DB commit (line 281), the row stays in status='PROCESSING' and payment.autoRefundStatus stays 'REFUND_PROCESSING_BANNED_USER'. Unlike the outbox drain (which has a 5-minute IN_FLIGHT->PENDING reclaim) and unlike embeddings (recoverStuckEmbeddings), there is NO mechanism that resets stuck PROCESSING refund_queue_items back to PENDING. releaseUnprocessedRows only handles the in-process time-budget break, not a crash. Result: the refund executed at Stripe but the system never records the Refund row, never updates the queue item, and never retries — a permanent reconciliation gap on a money-moving, abuse-resistance path (banned-user auto-refunds). The grep for PROCESSING recovery across src confirms refund_queue_items has no reclaim job.
  - **Fix:** Add a stale-PROCESSING reclaim for refund_queue_items mirroring the outbox IN_FLIGHT reset: at the start of processRefundQueueOnce (or in a maintenance task), UPDATE refund_queue_items SET status='PENDING', attempt_count=GREATEST(attempt_count-1,0) WHERE status='PROCESSING' AND updated_at < NOW() - INTERVAL '5 minutes'. The existing Stripe idempotencyKey (auto-refund:paymentId:reason) makes a re-run safe (no double refund), so reclaiming is the missing half of the at-least-once contract.
- **Owner DELETE escapes moderation: ADMIN_PAUSED/locked listings can be hard-deleted**  
  `src/app/api/listings/[id]/route.ts:416-464 (DELETE transaction); FOR UPDATE select at 418-428 omits statusReason; report-count branch at 434-463` · _listings-crud / authz_
  - Every other listing mutation (PATCH profile, PATCH availability, updateListingStatus, recoverHostManagedListing, admin status change) gates writes through getHostModerationWriteLockResult() so a host cannot modify a listing whose statusReason is ADMIN_PAUSED or SUPPRESSED. The owner DELETE transaction does NOT perform this check. It only branches on report.count: if reportCount > 0 it suppresses, otherwise it hard-deletes. An admin can set statusReason='ADMIN_PAUSED' WITHOUT creating any Report row (see admin.ts line 422-424, where PAUSED maps statusReason to ADMIN_PAUSED with no report dependency). Such a listing has reportCount === 0, so the owner can permanently DELETE it — destroying the moderation-paused content and the canonical inventory row — fully evading the moderation hold.
  - **Fix:** Inside the DELETE FOR UPDATE transaction, after loading the row, call getHostModerationWriteLockResult({ statusReason: listing.statusReason, moderationWriteLocksEnabled: features.moderationWriteLocks }) and, if locked, return 423 LISTING_LOCKED before deleting/suppressing. Select statusReason in the FOR UPDATE query (it currently selects only ownerId, images, version).
- **Hard-delete / report-suppression DELETE path bypasses the moderation write-lock**  
  `src/app/api/listings/[id]/route.ts:415-464 (DELETE transaction); missing guard analogous to PATCH lines 682-695 and 1103-1116` · _reliability-state / authz_
  - The host-managed availability PATCH and the profile PATCH both call getHostModerationWriteLockResult and return 423 LISTING_LOCKED when statusReason is ADMIN_PAUSED or SUPPRESSED (a listing a moderator froze). The DELETE handler does not: after locking the row FOR UPDATE and confirming ownership, it proceeds straight to either suppress (if reports>0) or hard-delete (tombstone + listing.delete) with no moderation-lock check. A host whose listing was ADMIN_PAUSED/SUPPRESSED by a moderator (e.g. under abuse investigation) can simply DELETE it, destroying the listing row and its evidence and tombstoning the canonical inventory, defeating the write-lock that the edit paths enforce. This is an abuse-resistance / auditability gap for exactly the listings under review.
  - **Fix:** In the DELETE transaction, after loading the locked row, call getHostModerationWriteLockResult({ statusReason: listing.statusReason, moderationWriteLocksEnabled: features.moderationWriteLocks }) and return 423 when it is set (admins/moderators take a separate path). Select statusReason in the FOR UPDATE query so it is available.
- **Contact-credit consumption: no test asserts the actual one-credit deduction (FREE/PACK/PASS create path)**  
  `src/__tests__/lib/payments/contact-paywall.test.ts:236,255,294,330 (all assert create NOT called); untested deduction paths at src/lib/payments/contact-paywall.ts:635-653 (PASS), 664-681 (FREE), 704-722 (PACK)` · _testing-gaps / correctness_
  - consumeContactEntitlement is the core money/abuse-resistance flow (paid user spends a credit to contact a host). The happy-path branches that actually consume a credit — PASS (contact-paywall.ts:635-653), FREE (:664-681), and PACK (:704-722), each calling createConsumption — are never asserted in the contact-paywall unit test. The only consume-path test ('does not consume twice for the same unit identity', line 220) forces findUnique to return an existing row so it short-circuits before any create. The 'source: FREE' reference (line 173) is in a read-only evaluate test, not a consume test. So there is no regression test proving a PACK purchaser is charged exactly one credit, nor that PACK selection picks a grant with remaining credits. A bug that consumed 0 or 2 credits, or selected an exhausted PACK grant, would pass CI.
  - **Fix:** Add tests for each consume path: (1) free user under limit -> contactConsumption.create called with source:'FREE'; (2) active PASS -> create with source:'PASS', entitlementGrantId set; (3) free exhausted + PACK with remaining credits -> create with source:'PACK' against the correct (oldest non-exhausted) grant; (4) free exhausted + PACK fully used -> ok:false PAYWALL_REQUIRED, no create.
